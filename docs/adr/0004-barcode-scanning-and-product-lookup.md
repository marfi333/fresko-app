# Barcode scanning and product lookup

We add a `barcode` column to **Products** to support scanning EAN-8/12/13 and UPC-A/E codes from a phone camera, and a two-step lookup pipeline that prefers an existing household Product over a public-database call. The primary driver is reducing friction in the Add-Entry flow: the most common reason a user opens the FAB is to record a purchase whose packaging carries a barcode, and re-typing the name every time is the worst part of the existing flow.

For the in-browser decoder we use [@zxing/browser](https://github.com/zxing-js/browser) restricted to the four food-relevant 1D formats. Alternatives considered: `html5-qrcode` (bigger bundle, no fine-grained format restriction) and `quagga2` (unmaintained). `@zxing/browser` is dynamically imported via `next/dynamic({ ssr: false })` from the FAB so the inventory route's bundle stays small until the user actually opens the scanner.

For the metadata lookup we proxy [Open Food Facts](https://world.openfoodfacts.org) through our own route handler `/api/barcode-lookup/[code]` rather than calling it directly from the client. Two reasons: (1) OFF asks API consumers to send an identifying `User-Agent: Fresko - Web - <version> - <origin> - scan` header (the browser blocks setting `User-Agent` from `fetch`), and (2) routing through the server keeps user IPs out of the public OFF logs. We apply a 5-second `AbortSignal.timeout` and map upstream failures to clean status codes (200 / 404 / 502 / 504) so the client never has to interpret OFF-specific error shapes.

The lookup pipeline is **local-first**: query `/api/products/by-barcode?code=…` (household-scoped) before touching OFF. The hook `useBarcodeLookup` returns a discriminated union (`existing` / `new-from-off` / `new-blank` / `error`) and the FAB threads that into `AddEntryDialog`'s pre-seeded `productChoice`. The user always lands in the same Add-Entry form regardless of outcome — manual entry remains the universal fallback when the camera is denied, the network is offline, or the barcode is unknown.

For uniqueness we use a **partial unique index** on `(household_id, barcode) WHERE barcode IS NOT NULL` rather than a global `UNIQUE NOT NULL` column. Different households legitimately share barcodes (e.g., two households both scan the same milk carton), and existing Products without a barcode must remain valid forever. The partial index gives us within-household duplicate prevention without forcing a backfill. Drizzle's `.where(sql\`barcode IS NOT NULL\`)` DSL emits this directly — no hand-edited migration needed.
